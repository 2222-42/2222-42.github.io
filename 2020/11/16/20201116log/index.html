joy

誰かを待つために、喫茶店で二時間ほど時間をつぶし、そこで本を読んでいた。

この時の体験から、人のために2時間も待ち続けることができるし、それが苦ではないことがわかった。

また、もし可能であれば、こういった喫茶店で時間をつぶすのを趣味にできればいいなと思う。

not good at female

純粋な人に慣れしていない。純粋な人、というのは、同僚や取引先、客と店主、というような文脈が付与されていない人のことであり、こういった人とコミュニケーションするのが苦手であることに気づいた。

広く言えば、友達がいないのではないだろうか。

time and money

人生を送るうえで、結局一番学ばねばらないのは、時間やお金の使い方である。そして、それを学ぶことと人とコミュニケーションをとることとは、相互反復、相互学習である。

その中で、結局、自分の時間やお金の使い方のスタイルと反れている人とはうまく仲良くなれないのではないかと思う。

同僚や学友と仲良くなりやすいのは、同じ時間と場所を共有して、時間とお金の使い方が似ているから、という説がありうる。

そのうえで、先日会った人への強烈な興味が一瞬でなくなった理由は、この部分での大きな違いがあったことに気づいたからかもしれないという仮説を建てる。

漢字

「裙」かと思ったが、この漢字には、人の性質を表すものはなく、衣服の性質を表すものである。

rust

TRPL pp.128-141

  • パスをpubキーワードで後悔する
    • pub mod ...
    • moduleは公開される
    • 中身は非公開
      • 中身もデフォルトで非公開なので、pubを付けて公開する
  • 相対パスをsuperから始めると
    • 親モジュールから始める相対パスになる
    • ..と同じ感じ
  • 構造体定義の前にpubをつけると
    • 構造体は公開される
    • フィールドは非公開のまま
      • 構造体はフィールドが公開されてなくとも不便ではない
      • フィールド名の前にpubをつければ公開にできる
        • 非公開フィールドがあるから、そのインスタンスを作成(construct)するときは、公開された関連関数が構造体によって提供されている必要がある。
          • これがないと非公開の値を指定できず、インスタンスを作成できない
  • enumの前にpubをつけると
    • enumも公開される
    • そのヴァリアントは全て公開される
      • enumの場合は、ヴァリアントが公開されていないと不便だから
  • useキーワード
    • パスを一度スコープに持ち込み、それ以降はパス内の要素がローカルにあるかのように呼び出すことができる
      • クレートルートで定義されているかのように使える
    • 絶対パスでも相対パスでもいい
    • useの慣例
      • 関数
        • 関数の親父モジュールをuseで持ち込む
          • フルパスを繰り返すことを抑えつつ
          • 関数がローカルで定義されていないこと、どこで定義されたのかを明確にできる
      • 構造体やenum
        • use フルパスと書くのｇあ慣例で、これは理由なしの自然発生
        • 同じ名前の2つの要素をuseでスコープに持つこむのは許されないので例外があり、その解決策は2つある
          • 親モジュールをuseで入れる
          • as を使ってリネームする
            • use fullpath as LocalName
    • pub use
      • 再公開(re-exporting)と呼ばれる
      • 要素を自分のスコープに持ち込む
      • 外部のコードがその要素を使える・呼び出せる
        • そのスコープに持ち込むことができるようになる
      • 使い道
        • ある構造でコードを書きつつ、別の構造で公開するということができる
        • そのコードの内部構造と、そのコードを呼び出すプログラマーたちのその領域に関する見方が異なるときに有用
          • (感想: よくわからん)
    • 外部ライブラリの使い方
      • cargo.tomlに依存先を追加する
      • Cargoがcrates.ioからダウンロードし、プロジェクトで使えるようCargoに命令する
      • useの行を追加し、自分のパッケージのスコープに持ち込む
      • 標準ライブラリも外部にあるクレートである
        • cargo.tomlに書く必要はないが、
        • useでその要素を外部クレートからパッケージのスコープに持ち込ませる必要がある
    • 同じクレートが同じモジュールで定義された複数の要素を使うとき、
      • 複数行にわけて書く方法以外に、
      • 同じ一連の要素をネスト下パスを使うことで、一行で、スコープに持ち込ませることができる
        • use (パスの共通部分):: {path1, path2}
          • 共通した部分が最初のパスに他ならない場合は、selfを使う。

isabelle

Concrete Semantics pp.160-162

  • conditionを除いてcommandが等しいことの三つの性質の証明
    • refl
    • sym
    • trans
  • 条件Pにおいて、commandが等しいことをいｓっているなら、preconditionを今日かすることで、statementを弱くすることができる
    • [| P |= c ~ c'; ALL s. P' s -> P s |] ==> P' |= c ~ c'
  • Congruence Rule(合同規則)について
    • それぞれに対し、等しいsub-commandが置き換えられる時に得られるようなもの
    • それぞれのルールに対し、commandやbool以外にside conditionがある
    • Sequential Composition
      • Pがinitial stateで満たされているなら、cの実行後の状態においてQは成り立つべきという、PとQとの関連の記述
    • IF
      • シンプルなルールを採用し、サイドコンディションなしであるが、これより強いような記述は可能
    • WHILE
      • コマンドの実行後において、Pが成り立つことを要求する
        • Pがloopにおけるinvariant(不変条件)であること
    • ALL s s'. approx t s --> (c, s) => s' = (fold c t, s) => s'を求める
      • 証明するためには4つのlemmaがある
      • [| (c, s) => s'; approx t s |] ==> approx(defs c t) s'
        • これの証明のためにはapprox_mergeapprox t1 s \/ approx t2 s ==> approx (merge t1 t2) sを証明する必要がある

3章

  • 人は植物
    • 必要なものは人それぞれ
    • 「興奮/退屈」と「自発的/注意散漫」のマトリックス
      • 自発的/注意散漫には、方向性が必要
        • 何をすべきかを理解し、構造化のスキルを身につけ、管理可能なタスクに分割すればいい
      • 興奮/退屈には、モチベーションが必要
        • モチベーションには二種類あり、一つは外発的
        • もう一つは内発的動機
          • 自律性
            • 方向性は与えられているとして、
            • 自分で考えて行動をする
              • これはプロダクトオーナーシップを感じる
          • 熟達
            • 新しいスキルを学び、既存のスキルを向上させるための時間をもつこと
              • これにより、鋭く、効率的で、効果的になる
          • 目的
            • プロダクトに対するフィードバックには目を光らせる
  • チームリーダーにならなくても、チームリーダーの行動を理解できるようになる
    • チームリーダーがどのパターンを使って、成功(失敗)につなげているかを見て
    • チームの原動力がより具体的に理解できる

4章

  • チームの外部の人たちとどうやり取りすればいいだろうか
    • チームのが文化を破壊するアウトサイダーから身を守る方法を学ぶ
  • 有害とは
    • ある種の振る舞いのことである
    • チームの文化に含めないこと・やりたくないこと
      • これらを明確化しておいた方がいい
        • 明確化しておくと、目に余る振舞いの特定とその批判ができるようになり、
        • 有害な振る舞いを排除する文化を作れる
    • 人が変わっても文化は残る
      • 強い文化は自己選択的である
      • 文化を強くし、残すため、そして、悪い方向に行かないようにするためには、すでに記述した強くする方法のおさらいをしよう
  • 脅威を特定する
    • 特にリスクが高いのはチームの注意と集中であり、これは数少ないリソース
      • HRTのない、意図的にいじわるするような、トロルがやってくる
      • そのような人が出てきたときに、取扱注意とこころにとめられるようにしよう
    • 古典的なパターンをいくつか
      • 他人の時間を尊重しない
        • 文書を読まない人だったりして、
        • 人のエネルギーを消費する
      • エゴ
        • 合意の決定を受け入れられない人
        • 異なる視点の意見に耳を傾けられない人
        • 妥協できない人
          • 正しさよりも結論にたどり着くかどうかが重要
      • 権利を与えすぎる
        • 要求すれど貢献はしない人たち
        • 権利を与えすぎるとトロルになる
      • 未熟なコミュニケーションと複雑なコミュニケーション
        • 本名を使わなかったりメディアによってニックネームを分けたりする
      • パラノイア
        • オープンで透明なコミュニケーション文化があったとしても、
        • 陰謀論を唱える被害妄想患者がいる
      • 完璧主義者
        • プロジェクトを停滞させる
  • 脅威を追い出す
    • 振る舞いを追い出す
      • 人の追放は必然的な結論ではない
    • いくつかの方法
      • 完璧主義者には別の方向性を与える
        • 完璧主義の対象となるテーマを変更する
        • 不平不満をいう人にも効果的で、テストや手戻りの確認を担当してもらうなどがある
      • トロルには黙殺し、餌を与えない
      • 感情的にならない
        • 仕事は優れたソフトウェアを書くことであり、
        • ご機嫌をとったり、正当化したりすることではない
        • 争いごとはきちんと選んで、平静にたもつようにしよう
      • 不機嫌の真実を探せ
        • 積極的に事実を探そう
          • 指摘されたことを好意的に解釈し
            • 悪口の部分を全部無視して
          • 真理をついているか、その人の経験から学べるか、などを確認する
      • 優しく追い出す
        • 平静を保って事実を見るのを応用するとできる
      • 諦める時をしる
        • どんなに振る舞いを直そうと頑張ってみても
        • 見込みがないと諦めて先に進まなければならないときがくる

          • 短期的にはチームの注意や集中を無駄にしても、長期的にはプロジェクトにメリットがあるか？
          • その衝突は有益な方法で解決できるだろうか
        • 短期的なメリットのために文化を妥協する必要はない
          • HRTの重要性を認めない頭のいいコントリビューターには気をつけろ
          • 技術的に貢献できる人は交換可能であり、

      • HRTに対する自分の期待を明確にすること
